Java-memory-area-and-memory-overflow-exception

Posted on 2019-03-21

运行时数据区域

Java虚拟机所管理的内存包括以下几个运行时数据区域。

Java虚拟机运行时数据区

程序计数器

程序计数器(Program Counter Register)是块较小的内存空间，功能与CPU中的PC一致。

各个线程拥有独立的PC，这部分区域为线程私有的内存。
如果线程正在执行的是一个Java方法，PC记录的是正在执行的字节码指令的地址；如果正在执行的是Native方法，这个计数器的值为空(Undefined)。

此内存区域是唯一一个在Java虚拟机规范中没有规定任何OutOfMemoryError情况的区域。

Java虚拟机栈

Java虚拟机栈（Java Virtual Machine Stacks）同样是线程私有的，生命周期与线程相同。

虚拟机栈描述的是Java方法执行的内存模型：每个方法在执行的同时都会创建一个栈帧（Stack Frame）用于存储局部变量、操作数栈、动态链接、方法出口等信息。每一个方法从调用直至执行完的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。

局部变量表存放了编译器可知的各种基本数据类型（boolean byte short int long double float char）、对象引用(Object reference)和returnAddress类型（指向一条字节码指令的地址）。

其中64位长度的long和double类型的数据占用两个局部变量空间（Slot），其余数据类型只占用一个。每个方法需要在帧中分配多大的局部变量是完全确定的，在方法运行期间不会改变局部变量表的大小。

在Java虚拟机规范中，对这个区域定义了两种异常状况：

  • 如果线程请求的栈深度大于虚拟机锁允许的深度，将抛出StackOverflowError异常。
  • 如果虚拟机可以动态扩展（虚拟机栈可以固定大小也可以扩展，当前大部分虚拟机支持动态扩展），如果扩展时无法申请到足够的内存，就会抛出OutOfMemoryError异常。

本地方法栈

本地方法栈（Native Method Stack）与虚拟机栈发挥的作用非常相似，区别不过是虚拟机栈执行Java方法服务，本地方法栈执行Native方法服务。

Chapter13-ExplicitLock

Posted on 2019-03-21

在Java 5.0之前，在协调对共享对象的访问时使用的机制只有synchronized和volatile。Java 5.0增加了一种新的机制：ReentrantLock。ReentrantLock并不是一种代替内置加锁的方法，而是当内置加锁机制不适用时，作为一种可选择的高级功能。

Lock与ReentrantLock

Lock

Lock接口，Lock接口提供了一种无条件的、可轮询的、定时的以及可中断的锁获取操作，所有加锁与解锁的方法都是显式的。

  • Lock的lock()方法是经常被使用的一个获取锁的方法，如果当前锁已经被占用，则等待。通过该方法获取的锁必须主动释放，发生异常时不会释放锁。通常情况，使用该方法是在try...catch块中，在finally中释放锁。
  • Lock的lockInterruptibly()方法能够中断正在等待获取锁的线程。如果当前锁已经被占用，则阻塞等待。在等待过程中调用interrupt()方法会中断当前线程的等待状态。值得注意的一点是：只有在等待的线程才会被中断，已经获取了锁的线程不会响应等待中断。
  • Lock的tryLock()和tryLock(long var1, TimeUnit var3)与前两个获取锁的方法不同的是：这两个方法有返回值，该返回值描述了获取状态。如果成功获取锁，返回true，否则返回false。也就是说，无论是否获取到锁，都会返回，不会阻塞。后一个方法与前一个方法不同的是，如果获取锁失败，后一个方法会等待一段时间，如果在期限内没有获取到锁，则返回false，同时该方法可以响应中断。

ReentrantLock

ReentrantLock实现了Lock接口，并提供了与synchronized相同的互斥性与内存可见性。同事ReentrantLock提供了可重入的加锁语义。

ReentrantLock是唯一实现了Lock接口的类。

轮询锁与定时锁

以下以转账的过程举例：转账需要获取两个账户的锁，如果执行顺序不正确（A线程获取了账户1的锁，尝试获取账户2的锁，B线程获取了账户2的锁，尝试获取账户1的锁）可能会导致死锁，为了避免死锁，轮询锁在不能同时获取两个锁时，释放已经获取的锁，并等待一段时间后再此尝试，在询问固定次数后再返回失败，使得线程能够平滑结束。

Java-NIO

什么是IO

IO即输入输出，是计算机与设备进行资源交换的出入口。在Java中，传统的IO以流的方式处理IO，所有的IO都被视作单个字节的移动，通过stream对象一次移动一个字节。流IO负责把对象转换为字节，然后再将字节转换为对象。

什么是NIO

NIO即new IO，这个库是在JDK1.4中引入的。它在标准Java代码中提供了高速的面向块的IO操作。NIO与IO具有相同的作用和目的，但实现方式不同，NIO使用块为交换方式，所以NIO的效率比IO高得多。

Java的API提供了两套NIO：

  • 标准输入输出NIO
  • 网络编程NIO

并发编程

并发是为了提升程序的执行速度，但并不是多线程一定比单线程高效，而且并发编程容易出错。若要实现争取且高效的并发，就要在开发过程中时刻注意以下三个问题：

  • 上下文切换
  • 死锁
  • 资源限制

上下文切换会带来额外的开销

线程的运行机制

  • 一个CPU每个时刻只能执行一条线程
  • 操作系统给每条线程分配不同长度的时间片
  • 操作系统会从一堆线程中随机选取一条来执行
  • 每条线程用完自己的时间片后，即使任务还没完成，操作系统也会剥夺它的执行权，让另一条线程执行。

openvpn原理

openvpn通过使用公开秘钥（非对称秘钥，加密解密使用不同的key，一个称为Public key，另外一个称为Private key）对数据进行加密。这种方式称为TLS加密。

openvpn使用TSL加密方式的工作过程是，首先server与client使用想同的CA证书，双方通过交换证书的方式验证双方的合法性，用于决定是否建立VPN连接。

然后使用对方的CA证书，将数据加密后发送给对方，由于使用的是对方CA证书加密，所以只有对方CA证书的Private key才能够解密数据，这就保证了此秘钥的安全性，并且该秘钥是定期改变的，对于窃听者来说，可能还没有破解出此秘钥，VPN通信双方可能就已经更换秘钥了。
